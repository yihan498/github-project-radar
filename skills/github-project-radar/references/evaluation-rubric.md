# 项目筛选与成稿评价体系

## 使用原则

先过硬门槛，再比较分数。每项采用 `0–5` 原始分，只能使用整数；将原始分除以 5 后乘以权重得到加权分。所有评分都要附一至三条证据路径或数据，并写一句反证/不确定性。没有证据时最高只能给 1 分。

同一项目由同一版本量表评价。不得因“今天没有更好的项目”放宽评分。没有项目达到 75 分时，交付候选简报，不写正式推荐。

## 候选项目硬门槛

以下任一项失败，即 `pass=false`：

1. 仓库可公开访问且冷存成功；部分冷存只能写概览。
2. README/主文档足以解释目标，并至少找到两个关键实现或配置路径。
3. 许可证状态已识别；许可证缺失不等于不能分析，但必须标为风险，且不得建议直接复用代码。
4. 没有明显恶意、凭证窃取或主要依赖规避授权的迹象；无法判断时标为未验证。
5. 未在近 30 日正式推荐过同一仓库，除非是明确的重大版本复评。
6. 热度与增长表述有本地观测支持；首次观测不得声称增长。

## 候选评分（100分）

### A. 问题价值与定位清晰度（15）

- 0：无法确认解决什么问题。
- 1：只有口号，用户、场景和边界模糊。
- 3：用户、任务、输入输出和不解决的问题基本明确。
- 5：问题重要，定位清楚，并能由文档与实际工作流交叉验证。

### B. 技术与产品实质（15）

- 0：空仓库、链接集合或无法运行的占位内容。
- 1：主要是包装或薄胶水，缺乏可分析机制。
- 3：存在两个以上有意义的模块、状态流或工程权衡。
- 5：核心机制完整、有明确边界，源码足以进行深入架构分析。

### C. 可借鉴性（15）

- 0：没有可迁移经验。
- 1：只能复述功能。
- 3：至少三个可迁移设计点，适用条件可说明。
- 5：对产品、架构、工程或研究方法有高复用价值，且能同时指出不宜照搬处。

### D. 证据充分性与可审计性（15）

- 0：只有标题或简介。
- 1：只有 README 宣传材料。
- 3：README/文档与源码/配置可交叉验证，提交 SHA 已保存。
- 5：另有测试、发布记录、设计文档、评测数据或真实案例，关键主张可追溯。

### E. 工程质量与维护信号（10）

- 0：明显不可用或高风险。
- 1：结构混乱、无版本/测试/维护信息。
- 3：结构、错误处理、测试或发布纪律中至少两项可靠。
- 5：工程约束完整，并有持续维护、兼容性或回归证据。

### F. 新颖性或时效价值（10）

- 0：重复常见方案且无新增价值。
- 1：主要是旧方案换壳。
- 3：组合方式、使用场景或近期版本有明确新意。
- 5：出现可验证的新范式、新机制或重大版本变化，值得现在讨论。

### G. 热度与社区信号质量（8）

- 0：热度信息不可得或明显操纵。
- 1：只有静态 star 总数。
- 3：活跃提交、fork、issue/release 或多日 star 中有两个信号一致。
- 5：多日增长、贡献者活动与版本/讨论形成相互支持的趋势。

### H. 成熟度与风险透明度（7）

- 0：高风险却无任何说明。
- 1：限制、许可证或安全边界模糊。
- 3：主要限制与使用边界可以识别。
- 5：项目主动公开失败、未验证项、迁移风险与安全边界。

### I. 对当前协作目标的适用性（5）

- 0：与当前 GitHub 雷达和内容研究目标无关。
- 1：只能作为一般资讯。
- 3：至少一个机制可用于改进发现、冷存、评价、写作或自动化。
- 5：存在两个以上可小步验证的改进点，并能给出成本与验收标准。

## 选择规则

先比较总分。分差不超过 3 分时，依次比较 D 证据充分性、I 当前适用性、近 30 日是否覆盖过同类型、最后比较 G 热度质量。不得以 star 总数直接破同分。

## 成稿质量评分（100分）

- 原内容忠实度与边界（20）：准确区分项目声明、代码事实、推断和缺失。
- 关键机制拆解深度（20）：至少三个机制，解释数据流、权衡与失败模式。
- 证据可追溯性（15）：重要结论有路径/链接，包含提交 SHA 和冷存日期。
- 批判性与风险平衡（10）：限制具体，不以热度代替质量。
- 通用经验提炼（15）：说明适用条件、反例和不宜照搬处。
- 定向提升建议（15）：与当前协作目标相连，包含成本、优先级和验收标准。
- 结构与表达（5）：结论先行、少重复、篇幅达标且无原始标记残留。

总分至少 85。任何硬失败优先于总分。
