# 深度分析稿规范

## 目标

写一篇以仓库一手材料为依据、约 8,000–12,000 个中文字符的项目拆解。重点回答“做了什么、怎样做、为什么值得关注、哪些设计值得借鉴”，而不是复述宣传语。

## 建议结构

1. 标题、项目链接、分析日期、冷存版本、证据充分性。
2. 先给结论：适合谁、解决什么问题、推荐强度、最大亮点与主要限制。
3. 项目缘起与问题边界：明确它解决和不解决什么。
4. 用户视角的核心能力：用具体工作流解释输入、过程、输出。
5. 仓库结构地图：入口、核心模块、配置、扩展点、测试、文档和发布链路。
6. 关键实现拆解：至少选择 3 个真正重要的机制，引用仓库相对路径并解释设计权衡。
7. 技术架构与数据流：语言、框架、依赖、状态、接口、持久化、部署。
8. 上手与验证：安装运行路径、最小示例、维护者声明与实际代码是否一致。
9. 值得借鉴之处：产品、架构、工程、文档、社区或商业化层面，说明适用条件。
10. 针对当前 GitHub 雷达的提升建议：把借鉴点转成可执行选项，而不是泛泛而谈。
11. 不宜照搬之处：复杂度、依赖、成熟度、安全、许可证、性能或维护风险。
12. 同类定位：只在证据足够时比较，避免泛泛罗列竞品。
13. 最终评价：给出观察清单和后续跟踪指标。
14. 证据索引：列出 README、关键源码、配置、发布页、提交或 issue 链接。

## 约束与检查

- 用“仓库显示”“据此推断”“建议”区分事实、推断和建议。
- Star 上升只能引用本地多日观测，写清区间和绝对增量；首日不得使用“暴涨”。
- 不把缺失文件直接写成“没有”；证据不足时停止扩写并降级为概览。
- 重要结论至少由 README/文档与源码/配置两类材料交叉支撑。
- 完稿必须写明冷存日期、提交 SHA、三个关键实现路径、限制和后续验证项。
- 定向建议必须引用本地 `data/profile/collaboration-profile.md` 或当前对话中的已知目标；公开模板不能替代真实用户目标，不猜测身份、行业或未表达的偏好。
- 每条建议使用统一字段：`当前情境`、`可迁移思路`、`建议动作`、`预期收益`、`成本与风险`、`优先级`、`置信度`、`验收标准`、`建议结论`。
- “建议结论”只允许 `采纳`、`小范围试验`、`暂缓`、`不建议`；结论是分析建议，不替用户做最终决定。

## 成稿质量门槛

按 `evaluation-rubric.md` 的成稿量表逐项打分并保存到 scorecard。总分至少 85/100，且不得触发硬失败：来源不足却写成长文、增长断言无多日数据、核心实现无路径证据、缺少限制、定向建议没有成本/验收标准、把缺失材料写成否定事实。
