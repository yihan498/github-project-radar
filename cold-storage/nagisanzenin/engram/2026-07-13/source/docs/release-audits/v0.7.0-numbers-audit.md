# v0.7.0 — The Numbers Audit (RELEASE_PROTOCOL §4.8)

Five questions, in writing, for every number this release adds or changes. This is the gate that
exists for bug class #1: *a number that is wrong in the direction that reassures the learner.*

**Running it found three defects that every other gate had passed over.** All three are recorded
here, and all three are fixed.

The numbers under audit: `qwk`, `qwk_min_run`, `exact_agreement`, `leniency_bias`, `test_retest`,
`confusion`, `direction`, `by_case_type`, `coverage.*`, `grader_unvalidated`, `verdict`.

---

## Q1 · Is it cross-consistent with every other number the engine computes?

**Yes — verified by running them side by side, which is the only way this question has ever been
answered honestly here.** (v0.6.2 shipped `retention` saying *100% recall* while `decay` said
*56% and falling*, on one state, because nobody had ever run both.)

```
grader-health        verdict=pass  qwk=0.93  unvalidated=False
stats.grader_health  verdict=pass  qwk=0.93  unvalidated=False
stats.retention      verdict=pass            unvalidated=False
retention (cmd)      verdict=pass            unvalidated=False
```

All four agree **by construction**, not by coincidence: `compute_grader_health()` is a single
source and `compute_retention()` stamps itself, so every caller — `stats`, `cmd_retention`, the
dashboard — reads the identical verdict. This is the v0.6.4 lesson applied up front: *a rule
implemented in four places is a rule wrong in three.*

`confusion` totals **198** while `n` reports **66**. Those reconcile as `n × runs` (66 × 3), and
`direction.judgments` now states it outright — see Q3, where the unlabelled version of exactly
this was a defect.

## Q2 · Which direction does it fail in?

| Number | Can it fail optimistically (flattering the learner)? | Verdict |
|---|---|---|
| `grader_unvalidated` | **No, structurally.** It is `true` until an audit says otherwise — *including when no audit has ever run*, and including when the latest audit file is corrupt. It fails toward *"we don't know"*, never toward *"it's fine"*. | Safe |
| `verdict` | **No.** `pass` requires n ≥ 30 **and** complete coverage **and** the teeth clear **and** ≥ 3 runs **and** qwk ≥ 0.70. Every gate fails closed. | Safe |
| `test_retest` | **No — it can never certify on its own.** Above 0.95 the engine *demands* leniency be strictly under the ceiling, and fewer than three runs cannot pass at all. | Safe |
| `qwk` | **Yes — and this is the residual risk of the release.** See below. | ⚠ |
| `leniency_bias` | **Yes**, by the same mechanism as `qwk`. | ⚠ |

**THE RESIDUAL RISK, stated plainly because it will not go away by being unstated:** the gold set
is **authored, not independently human-adjudicated**. If the adjudicator (a model) shares the
grader's (a model's) blind spots, then QWK is inflated and the leniency bias is understated — and
*this audit cannot detect that*, because the thing it would need in order to detect it is the very
thing it lacks. The 88%-adversarial construction pushes against this (the cases were built to be
where graders fail) and the measured result is *harsh*, not lenient, which is the opposite of the
direction shared-bias would produce. **But neither of those is proof, and getting a second human
through the 66 items is the single highest-value item on the parallel track.**

**The asymmetry is deliberate.** `leniency_bias > +0.15` fails; a *negative* bias does not, at any
magnitude. That is correct — an optimistic number gets believed and stops the learner reviewing,
while a pessimistic one only annoys them. It is also **not free**: a harsh grader makes a learner
re-drill concepts they had genuinely earned and depresses the north star. `by_case_type` is where
that shows, and it is why the read names the weakest one.

## Q3 · What is its denominator, and does it say so?

**⚠ DEFECT FOUND AND FIXED. This is the v0.6.4 unlabelled-denominator bug, reproduced inside the
release built to catch unlabelled denominators.**

`by_case_type` emitted `n: 30` for `fluent-but-empty` — a case type with **ten items**. The 30 was
*judgments* (10 items × 3 runs), and **nothing said so**. A reader looking at a 66-item gold set
and a case type reporting `n=30` would conclude there were 30 such items. There are ten.

- **Fixed:** `by_case_type` now emits `items` **and** `judgments` as separate, named keys. The
  ambiguous bare `n` is gone from that block entirely, and a selftest asserts it cannot come back.
- `confusion`'s counts are likewise judgments, and `direction.judgments` now publishes the total
  beside them so `198` and `66` reconcile on the page rather than in the reader's head.

**The one denominator that was right from the start** is the audit's own `n`: the set of gold items
graded in **every** run (the intersection), never the union. A grader that drops 20 of 66 sids and
nails the rest gets `incomplete`, never a flattering `QWK 1.00`. `coverage` publishes `gold_n`,
`measured`, the dropped sids **by name**, unknown sids, and malformed-gold count.

## Q4 · Does anything actually READ it?

**⚠ DEFECT FOUND AND FIXED. The single most important fact in the entire release was computed,
stored, and narrated by nothing.**

`confusion` held the answer to *"has this grader ever inflated a grade?"* — and no runtime surface
read it. The audit could report `leniency_bias: +0.00` for a **perfect** grader and for one that
inflates a third of the set and deflates another third. **Same mean. Opposite products.** One is
safe; the other quietly tells learners they know things they do not.

- **Fixed:** `direction: {graded_up, graded_down, exact, judgments}` — and it goes into the **`read`
  string**, not a nested key: *"of 198 judgments it graded UP 0 times (the only direction that can
  flatter) and DOWN 22."* A selftest builds two graders with the same near-zero mean bias and
  opposite direction profiles and demands the read tell them apart.

Everything else reaches a narrator: `grader_unvalidated` → `retention.read` (stamped) → `/coach`
(required to voice it before any retention figure); `coverage` → `reasons` → `read`; `by_case_type`
→ the weakest row is named in `read` whenever it is below the grader's own average.

## Q5 · Can it be reached from the CLI in a way the skills never take?

**⚠ DEFECT FOUND AND FIXED.**

`assessor-audit --gold F` lets **any file** be the ground truth. The skills always use the bundled
set — but the audit file recorded only `gold_n`, so a `pass` measured against a hand-made 30-item
gold set was **indistinguishable on disk** from a pass against the shipped adversarial one. The
whole meaning of a QWK is *which set it was measured against*, and the artifact did not say.

- **Fixed:** every audit records `gold_source` (`"bundled:gold/assessor-gold.jsonl"` or the absolute
  path of the override), and it survives into `grader-health`. A selftest asserts it.

The other CLI-reachable literals are validated at ingest: `grade` must be in `GRADES` or the item is
not counted; `sid` must be a non-empty string; a run that is not a list is refused; and a payload
carrying `gold_grade` **dies** rather than certify a contaminated audit.

## What this gate cost, and what it bought

Three defects, none of which the selftest (155 green), the mutation test (24/24 killed), the fuzz
gate (0 / 9,750), or the live test could see — because **each of those confirms what you already
believed when you wrote it.** Two of the three were the *same failure this project has now shipped
three times*: a number whose denominator is not on its label.

**The gate that finds bugs is the one that makes you write down what you think the number means.**
